Purpose and Rationale

The PanOps Platform was created to standardize distrubted systems resource management in order to implement instances of distributed applications. Prior to the creation of PanOps individuals or groups that wished to manage diverse sets of resources, both local or utility based, either had to invent ad-hoc systems built in house, or couple themselves to proprietary utility computing management platforms like Heroku, or Amazon's CloudFormation system. PanOps provides a freely available, consistently interfaced, and well integrated alternative to either in-house development or proprietary lock-in, as well as access to a rich toolset and a well tested, ready-built system.

What is PanOps?

The PanOps Platform is a reliable, operable, distributed, self-organizing, resource management platform for distributed applications.

Reliable

Reliability is the first guarantee of the PanOps platform. In relationship to the PanOps platform, reliability is best described as determinism. That is, when given a known state of managed resources and information about that knowledge, and a transformation from that state to another given state PanOps components should always converge on an identical resultant state. This component behavior provides predictability in operations, eases testing, and codifies expections for developer operators.

Operable

Operability refers to the ability of a system to keep its operators informed of its state and the information about that state in the past, present, and when known, future and to hold that state constant given changes in the state or availability of surrounding resources.

PanOps handles failure as a first class citizen. In any distributed system, failure is inevitable. PanOps handles failures in underlying resources by continuously monitoring and reacting to resource state changes, automatically reconverging the system to its goal state without human interaction. Only if PanOps is or may be unable to successfully automatically reconverge the system to the specified goal state is corrective action required.

PanOps components chronicle all actions, decisions, and considerations and their reasons in a standard format using a standard interface for retrieval so that if human intervention into system operation is required to debug a problem or correct an issue context may be quickly gathered and effective corrective action may be taken.
