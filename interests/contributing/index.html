Reporting an issue

PanOps uses GitHub Issue Tracking to track issues (primarily bugs and contributions of new code). If you've found a bug in PanOps, this is the place to start. You'll need to create a (free) GitHub account in order to either submit an issue, comment on them or create pull requests.

Bugs in the most recent released version of PanOps are likely to get the most attention. Also, the PanOps core team is always interested in feedback from those who can take the time to test edge PanOps. Later in this guide you'll find out how to get edge PanOps for testing.

Creating a bug report

If you've found a problem in PanOps which is not a security risk do a seach in GitHub Issues in case it was already reported. If you find no issue addressing it you can add a new one. (See the next section for reporting security issues.)

At the minimum, your issue preport needs a title and descriptive text. But that's only a minimum. You should include as much relevant information as possible. You need to at least post the code sample that has the issue. Even better is to include a unit test that shows how the expected behavior is not occurring. Your goal should be to make it easy for yourself -- and others -- to replicate the bug and figure out a fix.

Then don't get your hopes up. Unless you have a "Code Red, Mission Critical, The World is Coming to an End" kind of bug, you're creating the report in the hope that others with the same problem will be able to collaborate with you on solving it. Do not expect the issue report will automatically see any activity or that others will just to fix it. Creating an issue like this is mostly to help yourself start on the path of fixing the problem and for others to confirm it with a "I'm having this problem too" statement.

Special treatment for security issues

Please do not report security vulnerabilities with public GitHub issue reports. The PanOps security policy page details the procedure to follow for security issues.

What about feature requests?

Please don't put "Feature Request" items into GitHub Issues. If there's a new feature that you want to see added to PanOps, you'll need to write the code yourself -- or convince someone else to partner with you to write the code. Later in this guide you'll find detailed instructions for proposing a patch to PanOps. If you enter a wishlist item in GitHub Issues with no code, you can expect it to be marked 'invalid' as soon as it's reviewed.
